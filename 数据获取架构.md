## 多数据源数据获取整体架构

本文档说明本项目中两大核心脚本的整体架构与数据流：

- `多数据源获取链接.py`：负责**根据业务场景 + 区域信息**，构造高质量的搜索 Query（含 site 过滤、时间过滤、区域域名映射等），将“要查什么”变成一组搜索任务。
- `多数据源获取数据.py`：负责**根据上游返回的 URL 列表**，统一抓取网页/文档内容，进行解析、清洗、结构化汇总，产出 LLM 友好的 Markdown 文本。

两者组合起来：前者决定“查哪些地址”，后者负责“从这些地址里把干净的信息抠出来”。

## 二、`多数据源获取链接.py`：查询构造与区域映射（简要）

### 2.1 搜索策略配置

- 使用 `SEARCH_STRATEGY_CONFIG` 统一定义各搜索类型的：
  - `includes` / `excludes` site 规则；
  - `regional_patterns`：根据区域信息（如省份）拼接具体政府/机构域名；
  - 时间过滤规则（after/before）。
- 例如托育政策与区域规划维度下：
  - 国家级：`site:nhc.gov.cn`、`site:moe.gov.cn` 等；
  - 区域级：`"site:wjw.{scope}.gov.cn"`, `"site:{scope}.tjj.gov.cn"` 等。

### 2.2 省份域名映射与模糊匹配

为了解决“用户输入中文省份/地级市，但实际政府站点域名用拼音/缩写”的问题：

- 维护 `PROVINCE_DOMAIN_MAP`：
  - 仅 31 个核心省级行政区短名 → gov.cn 域名标识（如：`"湖北" -> "hubei"`, `"四川" -> "sc"`, `"黑龙江" -> "hlj"`, `"广西" -> "gxzf"`）。
- `_normalize_scope_to_domain(scope_value: str)`：
  - 若输入是英文小写（如 `hubei`），直接透传；
  - 否则对中文输入做**模糊匹配**：按 key 长度从长到短，判断 `"黑龙江" in "黑龙江省哈尔滨市"` 等，命中后返回对应 slug。
- 在搜索侧使用方式：
  - 在 `web_search()` 中预计算一份 `domain_regional_data`：
    - 只有在执行 **区域性检索任务（is_regional=True）** 时，才会把 `scope` 替换为域名标识；
    - 普通搜索（例如 `"保育员" AND "湖北" site:gov.cn`）保留中文原文。

通过这种方式，实现：

- `policy_regional` 等需要按区域限定政府站点的查询自动使用域名缩写；
- `exclusive_rules` 中的 `{scope}` 保持用户原始中文（做语义过滤），不受映射影响。

## 三、`多数据源获取数据.py`：输入解析与调度

### 3.1 输入解析 `_parse_input_data`

输入来源通常为上游代码节点或检索节点输出，核心逻辑：

- 支持两种结构：
  - 顶层直接是数据对象；
  - 顶层包了一层 `datas`。
- 解析出：
  - `web_url_info_list`：所有网页/文档 URL，结构包含：
    - `url`, `title`, `source`, `snippet`, `provider`, `query`, `origin_key`；
  - `video_url_info_list`：所有视频 URL 及视频相关字段；
  - `career_payload`：招聘查询所需的 `keywords`、`provinces`、`page_size` 等；
  - `enterprise_names`：天眼查要查询的企业名称列表。
- 按数据特征与 `run_mode` 推断当前模式：
  - `legacy`：兼容旧版结构（`comprehensive_data` 为主）；
  - `external/Tuoyu`：用于托育外部数据流程，分为 general / institution 等。

### 3.2 调度中心 `DataOrchestrator.process_all`

职责：根据不同输入，**条件化+并发**地调度所有任务：

- 使用一个共享的 `httpx.AsyncClient`（支持 HTTP/2、禁用证书校验等）。
- 内容抓取任务：
  - 对每个 `web_url_info`：
    - 根据 `provider` 字段选择对应 Scraper；
    - 若找不到则默认用 `SearchApiScraper`。
- 招聘任务：
  - 若 `career_payload` 中 `keywords` 和 `provinces` 都存在，则创建一条 ZhiLian API 任务；
  - 否则跳过并记录原因。
- 企业信息任务：
  - 若 `enterprise_names` 非空，为每个企业创建一条天眼查查询任务。
- 使用 `asyncio.gather` 并发执行所有任务，并按三块结果回填：
  - `content_results`（网页/文档内容）；
  - `job_result`（招聘信息）；
  - `enterprise_results`（企业列表信息）。

### 3.3 主入口 `main_async` / `main`

- `main_async`：
  - 解析输入 → 调用 `DataOrchestrator` → 整理输出结构：
    - 按 `origin_key` 把内容分别塞到：
      - `comprehensive_data.all_source_list`；
      - `general_web_data`；
      - `institution_source_data`。
    - 视频结果统一整理为 `all_video_list`。
    - 招聘与企业信息分别归并为 `career_postings` 与 `enterprise_infos`。
- `main`：
  - 在 Dify 中同步调用 `asyncio.run(main_async(...))`；
  - 使用 `_dify_debug_return` 在本地调试时把输出漂亮地打印出来，并返回给 Dify 前端。

## 四、统一文件解析服务 DocumentParserService

`DocumentParserService` 是一个**统一的文件解析与清洗中心**，将各种格式的二进制文件转为 LLM 友好的 Markdown 字符串。

支持的主要类型：

- PDF (`.pdf`)
- Word (`.docx`, `.doc`)
- PowerPoint (`.pptx`, `.ppt`)
- Excel (`.xlsx`, `.xls`)
- CSV (`.csv`)
- HTML 文件 (`.html`, `.htm`)
- JSON (`.json`)
- XML (`.xml`)
- 纯文本 (`.txt`)
- Markdown (`.md`, `.markdown`)
- 图片 (`.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`, `.bmp`)

### 4.1 核心调度：`parse(binary_content, file_extension, source_url)`

调度规则：

- 根据扩展名 `ext` 选择对应 `_parse_xxx`：
  - `.pdf` → `_parse_pdf`
  - `.docx` → `_parse_docx`
  - `.pptx` → `_parse_pptx`
  - `.xlsx/.xls` → `_parse_excel`
  - `.csv` → `_parse_csv`
  - `.html/.htm` → `_parse_html_file`
  - `.json` → `_parse_json`
  - `.xml` → `_parse_xml`
  - `.txt` → `_parse_plain_text`
  - `.md/.markdown` → `_parse_markdown`
  - 图片后缀 → `_parse_image`
  - `.doc/.ppt` 等老格式优先走 MarkItDown，再结合 `_upload_embedded_images` 和清洗。
- 若专用解析结果为空：
  - 使用 MarkItDown 做最后兜底；
  - 文档类格式再走一次图片抽取+上传逻辑；
  - 始终确保返回一段对 LLM 友好的 Markdown（或提示无法解析）。

### 4.2 文件类型解析细节

只列关键要点，完整逻辑见源码：

- **PDF `_parse_pdf`**：
  - 使用 PyMuPDF(fitz)：
    - 最多处理 `PDF_MAX_PAGES` 页；
    - 对每页按 block 的 y 坐标排序，抽取行文本；
    - 大图块转为 `![图片N (第P页, WxH)](pdf_image_N)` 占位符。
  - 调 `_upload_embedded_images(data, '.pdf', md)`：
    - 再次从 PDF 二进制中提取真实图片；
    - 使用 `EmbeddedImageUploader` 上传到你配置的图片服务；
    - 替换 Markdown 中的 base64 / 占位符为真实 URL。
  - 最终通过 `DataCleaningPipeline.clean_document` 去除页眉页脚、版权尾巴、页码、无意义符号等。

- **Word DOCX `_parse_docx`**：
  - 优先用 MarkItDown 一步转 Markdown；
  - 若失败且有 `python-docx`：
    - 遍历 paragraph：
      - Heading 段落转为 `#`/`##` 等；
      - 普通段落直接输出；
    - 遍历 table → `_rows_to_md_table` 转 Markdown 表格。
  - 之后同样走 `_upload_embedded_images` + `clean_document`。

- **PPTX `_parse_pptx`**：
  - 使用 `python-pptx`：
    - 按 slide 遍历 shapes：
      - 文本框（含标题占位符） → 文本内容；
      - 表格 → 行列转 Markdown 表；
      - 图片 shape → 截图 blob 和扩展名，加入上传列表，Markdown 中保留占位符。
    - 每页合成为 `## 幻灯片 N: 标题` 加正文列表。
  - 批量上传全部图片，替换占位符为真实 URL，最后 `clean_document`。

- **Excel `_parse_excel`**：
  - 优先用 MarkItDown 转 Markdown 表；
  - 再用 `openpyxl` / `xlrd` 读表：
    - 限制 `MAX_TABLE_ROWS` 行，超出尾部插入“已截断”说明；
    - 各 sheet 输出为 `### 工作表: Name` + Markdown 表；
  - 用 `clean_table` 规整空白 + 截断。

- **CSV `_parse_csv`**：
  - MarkItDown 兜底；
  - 否则手动用 `csv.Sniffer` / `csv.reader` 解析，最多 500 行，转 Markdown 表，再 `clean_table`。

- **HTML 文件 `_parse_html_file`**：
  - `_decode_bytes` → HTML 字符串；
  - `trafilatura.extract(..., output_format='markdown', include_images=True, favor_recall=True)` 提正文；
  - `clean_html` 去噪。

- **JSON `_parse_json`**：
  - `_decode_bytes` → 文本；若能 `json.loads` 成对象则 `indent=2` 美化；
  - 超出 `MAX_JSON_CHARS` 截断；
  - 包装为 Markdown 的 ```json 代码块。

- **XML `_parse_xml`**：
  - 优先 MarkItDown → `clean_text`；
  - 否则 `_decode_bytes` + 长度截断，包装为 ```xml 代码块。

- **TXT / Markdown**：
  - 解码后走 `clean_text`，只做空白规范化和长度截断。

- **图片 `_parse_image`**：
  - 若有 `source_url`，先输出 `![image](source_url)`；
  - 若安装 `PIL`，附加图片格式、尺寸、颜色模式说明；
  - 若安装 `pytesseract`，附加 OCR 文本段落。

## 五、数据清洗策略（去噪、截断、图片校验）

### 5.1 `DataCleaningPipeline`

适用于 DocumentParserService 输出的 Markdown / 文本：

- `_NOISY_PATTERNS`：
  - 过滤分隔线、URL/HTML 结尾行、门户版权/备案尾巴、`[下一页]`/`[上一页]`/`[编辑]`等无关链接、分享/二维码/责任编辑行等。
- `_remove_repeated_headers_footers`：
  - 统计短行出现次数，出现次数 ≥ 阈值（默认 3）视作页眉/页脚，整篇剔除。
- `_PAGE_NUM_PATTERN`：
  - 识别并移除 `第 N 页`、`Page i of j` 等页码行。
- `_normalize_whitespace`：
  - 归一化空白、多空行压缩为单空行。
- `_truncate`：
  - 统一截断过长内容（例如 80K 字符），尾部标注“已截断”说明。
- 针对不同场景的入口：
  - `clean_document`：文档类（PDF/DOCX/PPTX）；
  - `clean_html`：网页正文；
  - `clean_table`：表格 Markdown；
  - `clean_text`：通用纯文本。

### 5.2 图片 URL 校验

两种场景：

- `DataCleaningPipeline.validate_image_urls`：
  - 对 Markdown 中图片 URL 进行异步 HEAD 请求验证（最多 25 个），过滤死链图片。
- `SearchApiScraper._remove_invalid_images_async`：
  - 针对在线 HTML 提取结果的 Markdown，过滤失效图片引用，然后再做文本清洗。

## 六、招聘与企业信息抓取

### 6.1 招聘信息：`ZhiLianJobScraper`

- 通过配置好的 API 网关 `http://119.45.167.133:12906/api/scrape/zhilian` 抓取招聘数据：
  - 请求体包含 `keywords`、`provinces`、`page_size` 等字段；
  - 响应统一封装为：
    - `{"status": "success/failed/skipped", "data": [...], "message": "..."}`。
- 在 `DataOrchestrator.process_all` 中作为独立任务执行，结果写入 `job_result`，最终映射为 `career_postings`。

### 6.2 企业信息：`TianyanEnterpriseScraper`

- 使用天眼查开放 API 查询企业基础信息：
  - 请求参数 `{ "keyword": 企业名称 }`；
  - 成功时返回 `{"status": "success", "data": result, "message": reason}`。
- 对 `enterprise_names` 列表中每个名字创建一个任务，最后在 `main_async` 中统计成功/失败数量，汇总为：
  - `status`（success/failed/partial_success/skipped）；
  - `data`（成功数据列表）；
  - `failed_queries`（失败的企业及原因）；
  - `summary`（总览说明）。

## 七、输出数据结构总览（给下游 RAG / LLM 用）

最终对外暴露的关键字段：

- `scraped_datas`：
  - 在 legacy 模式：
    - `comprehensive_data.all_source_list`：所有网页/文档内容（按 URL 聚合），每条包含：
      - `type`, `source_id`, `url`, `title`, `source`, `snippet`, `query`, `content`(Markdown)。
    - `comprehensive_data.all_video_list`：视频条目列表；
    - `career_postings`：招聘抓取结果；
    - `enterprise_infos`：企业查询结果汇总。
  - 在 external/Tuoyu 模式：
    - `general_web_data`：通用 Web 内容；
    - `institution_source_data`：机构/官方来源内容；
    - `career_postings`：同上。
- `scraped_datas_str`：
  - `scraped_datas` 整体结构的 JSON 字符串，方便直接存入向量库或日志。

整体上，本架构实现了：

- 从“多搜索引擎 + 多数据源”的结果中，统一抽取网页 / 文档 / 表格 / PDF / 图片等内容；
- 自动进行解析、清洗、截断、图片上传与链接替换；
- 按来源与用途分组输出结构化数据，直接供 RAG/LLM 以及下游业务节点使用。
